Create a DNN Manifest

Overview

The DNN manifest is an XML file (e.g., MyDNNExtension.dnn) that specifies how each file in the extension package must be processed during installation.

Note: The file extension must be .dnn. You can add the DNN version at the end; e.g., MyDNNExtension.dnn8.

Steps

  1. Start with a basic manifest. <dotnetnuke type="Package" version="8.0"> <packages> <package name="MyCompany.SampleWidget" type="Module" version="1.0.0"> <friendlyName>My Sample Widget</friendlyName> <description>My Sample Widget is a demonstration widget.</description> <iconFile>"MyIcon.png"</iconFile> <!-- Are the quotes needed, just like license and releaseNotes? --> <owner> <name>MyCompany or MyName</name> <organization>MyCompany Corporation</organization> <url>www.example.com</url> <email>support@example.com</email> </owner> <license src="MyLicense.txt" /> <releaseNotes src="MyReleaseNotes.txt" /> <azureCompatible>true</azureCompatible> <dependencies> <!-- Insert dependencies here. --> </dependencies> <components> <!-- Insert details of components here. --> </components> </package> </packages> </dotnetnuke>

    Replace the appropriate values with your information:

    • package <package name="MyCompany.SampleWidget" type="Module" version="1.0.0"> ... </package>
      • name. Must be unique. To ensure your package's uniqueness, add your company as the prefix.
      • type. Can be one of the following:
        • Authentication System
        • Container
        • Dashboard Control
        • Language Pack: (CoreLanguagePack, ExtensionLanguagePack)
        • Library
        • Module
        • Provider
        • Skin
        • Skin Object
        • Widget
        • other custom extension types

      • version. The version of your extension.

      Each package represents a DNN extension. You can install multiple extensions using a single DNN manifest by creating a package section for each extension inside the packages tag.

      Note: The extensions are installed in the order they appear in the manifest.

    • friendlyName and description. The friendlyName can contain spaces and up to 250 characters. The description can hold up to 2000 characters.
      • <friendlyName>My Sample Widget</friendlyName> <description>My Sample Widget is a demonstration widget.</description>
    • (Optional) iconFile. The image file (preferably .png) to be associated with the extension. The icon is displayed in the DNN Control Panel's dropdown list and in the Extensions page. <iconFile>"MyIcon.png"</iconFile>

      Note:The iconFile section must be between the description and the owner sections.

    • (Optional, but encouraged) owner. Information about the owner or creator of the extension.
      • <owner> <name>MyCompany or MyName</name> <organization>MyCompany Corporation</organization> <url>www.example.com</url> <email>support@example.com</email> </owner>
    • (Optional, but encouraged) license and releaseNotes. These text files are displayed during the installation. The user is prompted to accept or decline the license. The text can also be embedded within the tag without the src attribute. <license src="MyLicense.txt" /> <releaseNotes src="MyReleaseNotes.txt" />
    • (Optional) azureCompatible. Set to true if the module is compatible with SQL Azure; otherwise, false. <azureCompatible>true</azureCompatible>

  2. Add dependencies. <dependencies> <dependency type="coreVersion">08.00.00</dependency> <!-- Insert additional dependencies here. --> </dependencies>

    Dependencies can be any of these types (case-insensitive):

    • coreVersion. Minimum DNN version required by the extensions being installed.
    • dependency list item
    • managedPackage
    • package. A package required by the extensions being installed. It must already listed in the core Packages table.
    • permission
    • type
  3. Add components. <components> <component type="Widget"> <widgetFiles> <basePath></basePath> <widgetFile> <path/> <name/> </widgetFile> </widgetFiles> </component> <!-- Insert additional components here. --> </components>

    A component type can be one of the following:

    Assembly
    AuthenticationSystem Authentication providers used by the module, such as OpenId, LiveId, and ActiveDirectory.
    Cleanup List of files that must be deleted when the module is uninstalled.
    Config
    Container DNN container
    DashboardControl
    File
    CoreLanguage
    ExtensionLanguage
    Module
    Provider
    ResourceFile
    Script
    SkinObject
    Skin
    URL Provider
    Widget
